## 一、背景与动机

在大语言模型和多模态模型时代，文档解析已成为预训练和检索的核心数据引擎，它决定了我们能从海量的 PDF、扫描件和截图中提取多少可靠的结构化监督信息。然而，传统 OCR 系统存在一个根本性局限：它们专注于文本识别，将图形区域（如图表、图标、流程图）简单裁剪为像素块并保留为栅格图像，导致文档中蕴含的结构和语义信息大量流失。

这一问题的核心在于：**文档不仅通过文字传达信息，更通过图形传递知识**。科学论文的图表、技术文档的流程图、网页的 UI 元素、化学结构的分子式——这些都是高度信息密集的视觉符号，传统 OCR 将它们视为"黑盒像素"而非"可理解的结构"。

本文提出的 **Multimodal OCR (MOCR)** 范式，正是为了解决这一痛点。它将视觉元素（图表、图表、表格、图标）提升为"一等公民"，将其解析为可渲染的结构化代码（如 SVG），而非保留为静态像素。这一转变不仅使文档解析更加忠实，更为多模态预训练解锁了一个全新的监督信号来源。

## 二、核心方法

### 2.1 任务定义

MOCR 的目标是对文档页面进行全面解析，包括 PDF 渲染、数字扫描、网页和场景文本图像。给定输入图像 **I**，任务是生成有序的解析元素序列 **S**：

**S = [(B₁, c₁, p₁), ..., (B_K, c_K, p_K)]**

其中：
- **B_k**：空间区域或边界框
- **c_k**：语义类别或元素类型
- **p_k**：关联的有效载荷

关键创新在于 **p_k** 的定义：对于文本区域，它是转录文本；对于视觉符号（UI 组件、图标、图表），它是**可渲染的结构化表示**（即图像到 SVG 的转换）。这使得 MOCR 能够实现"渲染-复用"工作流。

### 2.2 模型架构

dots.mocr 采用经典的视觉-语言模型架构，包含三个核心组件：

**（1）高分辨率视觉编码器（1.2B 参数）**
- 完全从零训练，原生优化文档解析任务
- 支持高达约 1100 万像素的输入分辨率
- 对于小字体文本和精细图形基元（图表标记、图表笔划）的精确感知至关重要

**（2）轻量级多模态连接器**
- 桥接视觉编码器和语言解码器

**（3）自回归语言解码器（Qwen2.5-1.5B）**
- 选择 1.5B 参数是容量与成本的权衡
- 需要同时处理异构页面内容（文本、布局、视觉符号）并生成 SVG 等长结构化输出

### 2.3 训练策略

采用三阶段渐进式预训练 + 指令微调的策略：

**阶段一**：建立稳定的视觉-语言接口，通过通用视觉训练让语言模型可靠地消费视觉标记

**阶段二**：在通用视觉数据和纯文本文档解析监督的统一混合上进行广泛预训练，建立强大的文本中心解析基础

**阶段三**：转向 MOCR 特定目标，减少通用视觉数据比例，增加多模态文档解析和图像到 SVG 的权重

**指令微调阶段**：使用数据引擎构建的高质量监督集，对 SVG 特定处理（规范化、viewBox 标准化、复杂度降低）进行优化

### 2.4 数据引擎

这是 MOCR 可扩展性的关键支撑，包含四大来源：

1. **PDF 文档**：使用 dots.ocr 自动标注引擎产生结构化页面转录，通过语言、领域和布局复杂度的分层采样强调困难场景

2. **网页**：爬取并渲染为页面图像，从 HTML/DOM 获取对齐的结构信号，提供原生的 SVG 图标和图表

3. **SVG 图形**：收集网络上的原生 SVG 资产，通过 svgo 清理、去重和复杂度感知采样

4. **通用数据**：保留通用视觉和 OCR 监督以维持广泛能力

## 三、实验设计

### 3.1 文档解析评估

采用 **Elo 竞技场** 框架（OCR Arena），使用 Gemini 3 Flash 作为评判者进行成对比较，通过 1000 次 bootstrap 重采样确保统计稳健性。相比传统的 WER/NED/TEDS 指标，这种 LLM-as-Judge 方案能更好地评估复杂 Markdown 输出的真实端到端质量。

### 3.2 结构化图形解析评估

在多个领域基准上评估：
- **UniSVG**：通用矢量图形
- **ChartMimic**：科学图表
- **Design2Code**：网页和 UI 布局
- **GenExam**：考试风格图表
- **SciGen**：科学插图
- **ChemDraw**：化学结构图

使用 **ISVGEN** 分数（渲染-比较协议）作为统一指标。

### 3.3 通用 VQA 评估

在 CharXiv、OCR Reasoning、InfoVQA、DocVQA、ChartQA、OCRBench、AI2D、CountBenchQA、RefCOCO 等基准上验证模型的通用能力。

## 四、结果与分析

### 4.1 文档解析性能

**OCR Arena Elo 排行榜**（三个基准平均）：
1. **Gemini 3 Pro**: 1210.7
2. **dots.mocr**: 1124.7 ⭐
3. dots.ocr: 1086.2
4. HunyuanOCR: 984.2
5. PaddleOCR-VL-1.5: 920.5

dots.mocr 在开源模型中排名第一，仅次于 Gemini 3 Pro。

**olmOCR-Bench 详细分解**：
- dots.mocr 达到 **83.9%** 整体准确率（新 SOTA）
- 在 ArXiv、Old scans math、Tables、Multi column 四个类别中取得最高分
- 证明其在学术论文、老旧扫描、复杂表格和多栏布局上的强大能力

### 4.2 结构化图形解析性能

**UniSVG Overall**：
- dots.mocr-svg: **0.902** ⭐
- dots.mocr: 0.894
- OCRVerse: 0.763
- Gemini 3 Pro: 0.735

**下游基准**：
- ChartMimic: 0.905 vs Gemini 0.788
- Design2Code: 0.834 vs Gemini 0.760
- ChemDraw: 0.901 vs Gemini 0.839

dots.mocr-svg 在所有报告的下游基准上都超越了 Gemini 3 Pro，这在 3B 参数规模下尤为惊人。

### 4.3 通用 VQA 能力

尽管专注于结构化解析，dots.mocr 在通用能力上依然保持竞争力：
- CharXiv Reasoning: **55.3**（显著超越 Qwen3-VL-4B 的 39.7）
- OCRBench: 86.0
- CountBenchQA: 94.46
- RefCOCO: 80.03

## 五、评价与思考

### 5.1 核心贡献

1. **范式创新**：首次系统性地将视觉符号提升为文档解析的一等公民，将"丢弃的像素"转化为"可复用的代码"

2. **工程实现**：构建了完整的可扩展数据引擎，通过规范化、去重和复杂度控制解决了非唯一程序目标的挑战

3. **评估框架**：提出的 OCR Arena（LLM-as-Judge + Elo 排名）为复杂文档解析提供了更鲁棒的评估方案

### 5.2 局限与挑战

1. **任务条件化**：当前版本尚未实现单次前向同时输出完整文档解析和 SVG 解析，需要分两轮执行

2. **参数规模**：3B 模型在某些复杂场景（如 GenExam、SciGen）上仍低于 Gemini 3 Pro，说明规模上限仍有空间

3. **领域覆盖**：目前聚焦于 SVG，未来需扩展到 TikZ、D3.js、CAD 等其他程序空间

### 5.3 深层意义

MOCR 代表了一个重要的范式转变：**从"文本中心 OCR"到"文档原生多模态解析"**。它认识到文档中最丰富的监督信号往往是视觉而非文本的——这正是传统 OCR 流程所丢弃的。通过将图形恢复为可执行、可重渲染的 SVG 程序，这项工作将静态文档像素转化为可用于推理和学习的结构化监督。

## 六、适用场景

### 6.1 立即可用

- **学术论文处理**：从 PDF 批量提取图表、公式、表格为结构化格式
- **知识库构建**：将企业文档转化为可检索的结构化数据
- **UI/UX 设计**：从截图自动生成前端代码（SVG）
- **化学/生物信息学**：从分子结构图提取 SMILES 等结构化表示

### 6.2 未来潜力

- **大规模预训练语料构建**：从海量文档中自动生成图像-代码-文本三元组
- **多模态 RAG 系统**：基于结构化内容的精准检索与推理
- **文档迁移与现代化**：将旧格式文档转换为现代可编辑格式
- **教育领域**：自动解析教材内容生成结构化题库

---

## 总结

MOCR 是文档解析领域的一次重要突破。它不仅解决了传统 OCR 忽视图形信息的痛点，更开辟了一条可扩展构建大规模多模态预训练语料的路径。dots.mocr 以仅 3B 的参数规模，在多项基准上超越了包括 Gemini 3 Pro 在内的强大竞争对手，证明了这一范式的有效性。

更重要的是，这项工作指明了文档智能的未来方向：**从读取文字到理解视觉语言**。正如论文结尾所言："我们希望这项工作能帮助推动文档解析从以文本为中心的 OCR，转向捕获人类知识完整视觉语言的文档原生多模态解析。"
