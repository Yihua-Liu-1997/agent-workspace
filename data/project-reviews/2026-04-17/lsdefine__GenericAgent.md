# GenericAgent: 基于3K行代码种子的极致轻量级自我进化Agent框架

## 项目简介

**GenericAgent** 是一个极简且具备自我进化能力的自主 Agent 框架。它通过极少的代码量（核心仅约 3000 行代码，核心的 Agent Loop 仅约 100 行），结合 9 个原子工具，赋予了大型语言模型（LLM）对本地计算机的完全系统级控制能力。这些能力涵盖了浏览器操作、终端执行、文件系统管理、键鼠输入、屏幕视觉乃至移动设备（ADB）控制。

该项目致力于解决当前主流 Agent 框架往往过于笨重、依赖繁杂且 Token 消耗巨大的问题。它的核心设计哲学是：“**不预设技能，靠进化获得能力**”。Agent 在解决新任务的过程中，会自动将探索出的执行路径和经验固化为可复用的技能（Skill），形成一棵随使用时长不断生长的“专属技能树”。

## 核心特性与技术架构

1. **极简架构与零负担部署**: 整个系统由约 3K 行核心代码构成，无复杂的第三方服务编排和繁重的环境依赖，仅需基础的 Python 环境即可本地启动，部署几乎“零负担”。
2. **分层记忆系统 (Layered Memory)**: 这是 GenericAgent 高效运作且极度节省 Token 的核心。其记忆被严谨地分为：
   - **L0 — 元规则**: Agent 的基础行为约束。
   - **L1 — 记忆索引**: 极简索引，用于快速路由。
   - **L2 — 全局事实**: 长期积累的稳定知识。
   - **L3 — 任务 Skills / SOPs**: 沉淀的特定任务解决流程。
   - **L4 — 会话归档**: 从已完成任务中提炼的长期记录。
   这种设计使得每次任务的关键信息得以持久化，跨会话积累经验，同时保持上下文窗口需求极低（<30K），极大降低了幻觉和 API 成本。
3. **最小工具集与动态扩展**: 出厂仅提供 9 个原子工具（如 `code_run`, `file_read`, `web_execute_js`, `ask_user` 等）。但通过核心的 `code_run` 工具，Agent 可以在运行时动态安装环境、编写脚本、调用外部接口，并将其临时能力固化为永久的新工具。
4. **自我进化机制**: 遇到全新任务时，Agent 会经历“自主摸索（安装依赖、调试代码） -> 成功验证 -> 提炼路径固化为 Skill -> 写入记忆层”的完整闭环。下次遇到同类任务时，只需一句话即可直接调用已沉淀的极简 Skill 脚本，实现了从“授人以鱼”到“授人以渔”的转变。
5. **真实的系统级控制**: 不同于纯沙盒或无头浏览器，GenericAgent 支持注入真实浏览器（可完美保留用户的登录态），并通过接管键鼠、视觉等真实物理输入输出，实现了极强的现实任务执行力。

## 适用场景与目标用户

**适用场景**:
* **个性化长尾任务自动化**: 适合处理那些没有现成软件支持、但对个人有高频需求的非标任务。如：定时量化选股并微信提醒、定期抓取特定网页生成报告、甚至自动化点外卖。
* **沉淀型私人助理**: 作为一个需要长期陪伴的私人智能助理，它会随着用户的个人习惯、特定项目需求不断“学习”和进化，越用越顺手。
* **跨应用复杂工作流**: 需要跨越浏览器、本地办公软件、终端甚至移动端（通过 ADB）的复合型任务处理。

**目标用户**:
* **极客与开发者**: 喜欢折腾、希望拥有一个完全由自己掌控、轻量且底层逻辑清晰的 Agent 框架的用户。
* **自动化办公探索者**: 经常需要处理繁琐、重复性系统级操作，且有一定动手能力的专业人士。

## 与同类项目对比

与目前市面上的重型 Agent 项目（如 OpenClaw 或 Claude Code 等体系）相比，GenericAgent 走的是一条截然不同的“极简+内生增长”路线：

| 特性维度 | GenericAgent | 主流重型 Agent (如 OpenClaw) |
| :--- | :--- | :--- |
| **代码体量与架构** | 约 3K 行，极简透明 | 动辄几十万行，依赖复杂的服务编排 |
| **工具箱生态** | 出厂仅 9 个原子工具，其余全靠运行生成 | 预设庞大的百种模块和插件生态 |
| **能力演进方式** | **自主生长**（运行时写代码沉淀技能） | **静态预设**（依赖开发者更新和社区插件） |
| **控制维度** | 真实浏览器注入、键鼠物理级控制、ADB | 偏向文件终端或沙盒无头浏览器 |
| **Token 消耗** | 极低（<30K），依赖分层记忆 | 通常较高（200K - 1M），依赖长上下文硬扛 |

## 评价与思考

GenericAgent 是一次对当前“重型化” AI Agent 发展趋势的精彩反思和反叛。当前许多框架倾向于在一开始就给大模型塞入大而全的工具库和复杂的 RAG 系统，导致系统臃肿、推理速度慢且成本高昂。

GenericAgent 的突破点在于它抓住了大语言模型时代的“第一性原理”：**LLM 的核心是强大的泛化推理和代码编写能力，而不是去生硬地记忆几百个预设 API 的参数**。

通过提供一个健壮的“执行-反馈循环（Loop）”和几个“基础物理动作（原子工具）”，结合优秀的分层记忆机制，GenericAgent 让 Agent 自己去写代码、造工具。这种 **“代码即工具，记忆即能力”** 的范式，不仅宣称将 Token 消耗降低了 6 倍，而且赋予了系统真正的、无限的扩展可能性。

该项目提到的“自举实证”（其 GitHub 仓库的维护本身就是由 GenericAgent 自主完成的）是对其自身能力逻辑闭环的最佳自证。

**潜在挑战**：这种高度自由的系统控制也带来了潜在的安全风险。执行未知任务时的“自主摸索”过程（包括执行生成的任意代码），可能会对本地文件系统造成不可逆的影响。安全性、操作边界的划定，以及初期形成优质 Skill 的失败率，是这类自我进化 Agent 必须面对的课题。

## 上手建议

1. **环境隔离（强烈建议）**: 由于其具备极高的系统权限和执行任意代码的能力，初次尝试强烈建议在一个独立的虚拟机、Docker 容器或不包含重要资料的测试机中运行，以防 Agent 在“摸索”阶段误删文件或破坏环境。
2. **选择顶级模型**: GenericAgent 的核心在于“写代码解决问题”，因此极度依赖模型的逻辑推理和编程能力。建议配置 Claude 3.5 Sonnet 等目前最顶级的代码模型，以保证早期“技能树”构建的成功率。
3. **循序渐进的“调教”**: 不要一开始就布置极其复杂的跨端任务。先从简单的单步骤任务（如：读取某目录下的所有TXT文件并总结）开始，让它成功生成几个基础 Skill，观察其记忆写入机制，然后再逐步布置需要串联多步骤的复杂需求。
4. **接入日常前端**: 项目提供了丰富的 Bot 接口。在终端跑通后，建议尽快将其接入个人微信、钉钉或 Telegram，将其从“命令行工具”真正转变为一个“随时在线的私人管家”。
