## 背景与动机

软件开发从来不是一锤子买卖，而是一个反复迭代的过程。然而，当前主流的编程 Agent 基准测试——从 HumanEval 到 SWE-Bench——几乎都是**单次评估（single-shot）**的：给一个完整的任务规格说明，跑一遍测试，看通过率。这种范式衡量的是"Agent 能不能一次性写出正确的代码"，却完全忽略了真实开发中更致命的问题：**写出来的代码好不好维护、好不好扩展**。

论文中提出了一个非常尖锐的观察：代码可以完美通过测试套件，但同时在结构性上变得日益难以维护和扩展。这种现象被作者形象地称为 "slop"——一种冗余、臃肿、结构腐化的代码积累。在传统软件工程中，这种"技术债务"的累积会导致维护成本指数级上升，而单次通过率基准完全捕获不到这个维度。

虽然近期也出现了一些迭代式基准（如 CodeFlowBench、EvoClaw），但它们要么为 Agent 提供了前一阶段的"标准答案"代码（从而破坏了设计决策的因果链），要么仅衡量通过/失败而忽略代码质量轨迹。SlopCodeBench 的核心目标就是填补这一空白：**让 Agent 在自己的代码上反复迭代，量化追踪代码质量如何随时间劣化**。

## 核心方法

### 基准设计原则

SlopCodeBench 提出了四条关键设计原则，确保能真实测量迭代开发中的设计决策影响：

1. **不预设内部接口**：只规定外部行为契约（CLI 参数或 API I/O），内部架构完全由 Agent 自行决定。
2. **不暴露测试套件**：Agent 只能看到自然语言规格说明和示例，不能获得测试反馈，必须自行推断边界情况。
3. **黑盒、语言无关**：问题设计只约束可观察行为，不依赖特定语言或生态系统。
4. **Agent 继承自己的代码**：前一阶段的输出就是后一阶段的起点，没有"金标准"可以依赖。

### 两个核心质量指标

论文定义了两个可在每个检查点计算的轨迹级质量信号：

**结构侵蚀度（Structural Erosion）**：衡量复杂度质量在多大程度上集中在高复杂度函数中。具体来说，对每个可调用对象计算"质量" = 圈复杂度 × √代码行数，然后统计圈复杂度 > 10 的函数所占的总质量比例。当 Agent 在迭代中反复往同一个函数里塞新逻辑而不是创建新的函数时，侵蚀度就会上升。

**冗余度（Verbosity）**：衡量冗余和重复代码的增长率。结合了 137 条 AST-Grep 规则捕获的常见冗余模式和克隆代码检测，归一化后得到 [0,1] 区间的分数。

### 典型例子：code_search

论文以 `code_search`（一个类似 ast-grep 的源码搜索工具）贯穿全文。该问题包含 5 个检查点：

- C1：仅支持 Python 的精确匹配和正则匹配
- C2：添加 JavaScript 和 C++ 支持
- C3：基于 AST 的模式匹配与元变量捕获
- C4：选择器规则和自动修复
- C5：添加 Go、Rust 和 Java 支持

在 C1 中硬编码 Python 逻辑的 Agent，到了 C2 和 C5 将面临雪崩式的重写；而一开始就构建可扩展解析器接口的 Agent 则不会。这种**早期的设计决策如何影响后期的维护成本**，正是 SlopCodeBench 要测量的。

## 实验设计

### 基准规模

- **20 个问题，93 个检查点**，每个问题 3-8 个检查点
- 每个检查点配有一个隐藏的测试套件，包含核心测试、错误处理测试、功能测试和回归测试
- 当前实验报告 Python 轨道的结果

### 评估的模型

11 个模型，涵盖 Anthropic、OpenAI 和智谱的前沿模型：

| 模型 | 工具 | 版本 |
|------|------|------|
| Claude Sonnet 4.5 / 4.6 | Claude Code | 2.0.65 / 2.1.44 |
| Claude Opus 4.5 / 4.6 | Claude Code | 2.0.51 / 2.1.32 |
| GPT 5.1 Codex Max | Codex CLI | 0.65.0 |
| GPT 5.2 / 5.2 Codex | Codex CLI | 0.71.0 / 0.80.0 |
| GPT 5.3 Spark / 5.3 Codex | Codex CLI | 0.100.0 / 0.98.0 |
| GPT 5.4 | Codex CLI | 0.110.0 |
| GLM 4.7 | Claude Code | 2.0.76 |

### 环境与协议

- 每个检查点在独立的 Docker 容器中运行，非 root 用户
- 检查点之间仅工作目录延续，shell 历史和 Agent 会话数据全部重置
- 每个检查点限时 2 小时，不设轮次上限
- 使用最精简的 prompt（just-solve），仅要求保持 requirements.txt 和入口脚本

### Prompt 干预实验

额外测试了三种 prompt 策略的效果：
- **just-solve**（基线）：仅要求实现规格
- **anti_slop**：明确禁止冗余模式、防御式过度工程和不必要的抽象
- **plan_first**：要求 Agent 先规划再编码

## 结果与分析

### 1. 没有任何 Agent 能端到端解决问题

**没有一个 Agent 能在任何问题上通过所有检查点**。最高严格通过率仅为 17.2%（Opus 4.6），隔离通过率（排除回归测试）在 9.7%–23.7% 之间，核心通过率在 26.9%–53.8% 之间。随着检查点推进，核心通过率和严格通过率之间的差距从 1.4 倍扩大到 13.3 倍——代码还能跑，但回归测试不断崩溃。

### 2. 质量持续劣化

- **侵蚀度**在 80% 的轨迹中上升，**冗余度**在 89.8% 的轨迹中上升
- 高复杂度函数的数量从平均 4.1 个增长到 37.0 个
- 最大圈复杂度从 27.1 飙升到 68.2
- 以 `circuit_eval` 为例，Opus 4.6 的 `main()` 函数从 84 行、CC=29 膨胀到 1099 行、CC=285

### 3. 与人类代码的标定对比

对比 48 个维护中的开源 Python 仓库：
- Agent 代码的冗余度（0.32）是人类代码（0.11）的 **2.2 倍**
- Agent 代码的侵蚀度（0.68）远高于人类代码（0.31）
- 追踪 20 个仓库的 git 历史（568 个时间检查点）：人类代码的冗余度和侵蚀度基本保持平稳，而 Agent 代码每次迭代都在恶化
- 即使是 scikit-learn（0.411）和 scipy（0.457）这样的复杂项目，侵蚀度也远低于 Agent 的平均水平

### 4. Prompt 干预无法阻止劣化

- anti_slop 能将初始冗余度降低 34.5%，plan_first 也能改善初始质量
- **但劣化速率（斜率）完全不变**——起点更低，但以同样的速度恶化
- 通过率（严格、隔离、核心）在所有 prompt 策略下均无统计显著差异
- 更"干净"的代码还带来更高的成本：anti_slop 在 GPT 5.4 上增加了 47.9% 的开销

### 5. 成本在增长，效果在下降

每个检查点的平均成本从第一阶段到最后阶段增长了 2.9 倍，但额外的投入并没有带来更好的正确性。

## 评价与思考

这篇论文在方法和洞察上都极为出色，我认为它触及了当前 AI 编程领域一个被严重低估的核心问题。

**首先，"slop" 这个概念非常精准。** 它不是简单的"代码写得差"，而是一种系统性的、在迭代过程中不可逆的劣化模式。这让人联想到软件工程中经典的"软件衰老"理论（Parnas, 1994）和技术债务概念，但发生在 Agent 代码中时，其速度和规模远超人类项目。Agent 代码像是一辆没有刹车的车——初始质量可以调优，但一旦开始迭代，劣化就无法停止。

**其次，实验设计非常扎实。** 不暴露测试套件、不预设内部接口、Agent 必须继承自己的代码——这些设计原则看似简单，却精确地切中了现有基准的软肋。用 48 个人类仓库做标定对比，更是为"劣化"这个概念提供了强有力的实证锚点。尤其是追踪 git 历史发现人类代码基本平稳而 Agent 代码持续恶化的结论，令人印象深刻。

**prompt 干预实验的结论尤其发人深省。** 它告诉我们，问题的根源不在于 Agent "不知道"应该怎么写好代码——anti_slop prompt 证明了 Agent 在被告知规则后能写出更好的初始代码——而在于**迭代过程中的结构性决策累积是一个本质性的难题**。这暗示了可能的解决方向：不是在 prompt 层面，而是在工具链层面（如自动重构、架构检查点）或训练层面（如在多轮迭代数据上训练）寻求突破。

**一个值得注意的细节**：论文使用的是各模型原生的 CLI 工具（Claude Code、Codex CLI），而非通用的 Agent 框架（如 SWEAgent）。这意味着这些结果反映的是真实开发者的使用体验，而非实验室里的理想情况，增强了结论的实用价值。

**当然也有一些局限性：** 目前只报告了 Python 轨道的结果；20 个问题虽然精心设计，但规模相对有限；标定对比用的是不同领域的仓库，严格来说并非"人类做同一任务"的对照实验。此外，侵蚀度和冗余度虽然能很好地捕捉结构退化，但是否就是最关键的质量维度，仍有讨论空间。

## 适用场景

1. **AI 编程工具的研发评估**：如果你在开发或评估 AI 编程助手，SlopCodeBench 提供了一个超越"通过率"的质量评估维度，特别适合评估长期迭代场景下的表现。

2. **代码生成模型的训练改进**：论文揭示了一个重要的训练信号——模型不仅要学会生成正确的代码，还要学会生成**可扩展、可维护**的代码架构。这为代码生成模型的 RLHF 或 SFT 数据构建提供了新方向。

3. **软件工程研究**：对于研究技术债务、代码质量演化的研究者，SlopCodeBench 提供了一个可控的实验平台，可以系统性地研究 AI 代码的劣化机制。

4. **AI Agent 安全与可靠性**：在企业环境中部署 AI 编程工具时，理解其长期代码质量轨迹对于评估技术风险至关重要。SlopCodeBench 的指标可以直接用于 CI/CD 流水线中的质量门禁。

5. **编程 Agent 的 benchmark 设计**：对于未来设计新基准的研究者，SlopCodeBench 的四条设计原则提供了一个优秀的方法论模板——特别是在如何隔离和测量迭代累积效应方面。
