# OccuBench: Evaluating AI Agents on Real-World Professional Tasks via Language Environment Simulation
### OccuBench: 通过语言环境模拟评估 AI Agent 在真实专业任务中的表现

## 背景与动机
随着大语言模型（LLM）能力的提升，AI Agent 被寄予厚望，期望其能深入各行各业执行专业任务（如急诊分诊、核反应堆安全监测、海关报关处理等）。然而，当前的 Agent 评测基准存在三大局限性：
1. **测试盲区（The Untestable Majority）**：现有的基准（如 WebArena, OSWorld, SWE-bench）大多局限于网页、操作系统、代码等公开环境。而真正高价值的专业领域（医疗、金融、制造）往往涉及私有系统，缺乏公开的 API 或交互环境。
2. **极高的扩展成本（Prohibitive Scaling Cost）**：为每个专业领域手动构建真实的沙盒环境或集成 API 成本极其高昂，难以覆盖成百上千的细分职业。
3. **缺乏鲁棒性评估（No Robustness Evaluation）**：现有评测多处于“理想路径（Happy Path）”，忽略了真实世界中 API 超时、数据丢失或静默降级等噪声环境。

为了填补这一空白，OccuBench 提出了“**将环境本身建模为语言模型**”的思路，通过 LLM 模拟复杂的专业环境。

## 核心方法

### 1. 语言环境模拟器 (Language Environment Simulator, LES)
LES 是 OccuBench 的核心创新。它将环境定义为一个黑盒函数：$(s_{t+1}, o_{t+1}) = f_\theta(s_t, a_t; c)$。
- **环境配置项 (c)**：包含系统提示词（业务逻辑规则）、工具定义（API Schema）、初始状态（如病房剩余床位）和状态描述（各字段的语义）。
- **工作原理**：模拟器利用 LLM 强大的领域知识，根据 Agent 的操作（$a_t$）和当前隐状态（$s_t$），生成逻辑一致的反馈（$o_{t+1}$）并更新环境状态。这使得评测不再依赖真实的软件后端。

### 2. 多智能体合成流水线 (Multi-Agent Synthesis Pipeline)
为了确保评测案例的质量，研究团队设计了一套自动化的合成流程，通过多个 Agent 协作生成任务实例，并满足四个条件：
- **可解性 (Solvable)**：通过模拟运行验证任务路径存在。
- **可验证性 (Verifiable)**：具备明确的成功指标和评估准则。
- **区分度 (Discriminative)**：任务难度经过校准，能拉开模型差距。
- **多样性 (Diverse)**：基于真实行业文档生成，覆盖 10 大行业类别的 65 个细分领域。

### 3. 环境故障注入 (Fault Injection)
OccuBench 模拟了四种环境状态，测试 Agent 的生存能力：
- **E0 (Clean)**：正常环境，基准性能。
- **E1 (Explicit Faults)**：显式错误（如 HTTP 500、超时、连接被拒）。
- **E2 (Implicit Faults)**：隐式错误（如返回数据被截断、字段缺失、数据陈旧，但无报错信息）。
- **E3 (Mixed)**：混合显式和隐式故障。

## 实验设计
研究者对 15 个前沿模型系列（包括 **GPT-5.2**, **Gemini 3.1 Pro**, **Claude Opus 4.6**, **Qwen 3.5 Plus**, **DeepSeek V3.2**, **Kimi K2.5** 等）进行了大规模评测。
- **覆盖范围**：100 个真实职业场景，382 个评测实例。涵盖 Business, Tech, Industrial, Transportation, Commerce, Education, Healthcare, Public Service, Agriculture, Science 十大类。
- **评估指标**：任务完成率（Completion Rate, CR）和鲁棒性得分（Robustness Score, R，反映受故障影响后的性能保持能力）。
- **模拟器选择**：默认使用 Gemini-3-Flash-Preview 作为 LES 的驱动模型。

---

## 结果与分析

1. **没有“全能型”模型**：实验发现，没有任何一个模型能在所有行业中统治。例如，**Gemini 3.1 Pro** 在教育（84%）和科学（81%）领域表现最佳，但在医疗（62%）领域不如 **Claude Opus 4.6**。每个模型都有其独特的“职业画像”。
2. **隐式故障是“ Agent 杀手”**：Agent 处理显式错误（E1）相对容易（通过重试），但在面对隐式故障（E2，如关键字段静默缺失）时，表现大幅下降（从 67.5% 跌至 53.4%）。大多数模型缺乏独立检测数据质量退化的能力。
3. **推理努力（Reasoning Effort）与规模的正向反馈**：更大的模型规模和更新的代际显著提升了专业任务表现。特别是 **GPT-5.2**，随着推理深度的增加（Thinking Depth），其性能从 54.7%（无推理）飙升至 82.2%（最高推理努力）。
4. **强 Agent $\neq$ 强模拟器**：研究发现，性能最强的模型（如 GPT-5.2）在作为 LES 模拟器时，并不一定比轻量级模型更稳定。模拟器的可靠性（如不捏造状态、不遗漏实体、严格遵循规则）是基于 LES 评估的核心前提。

## 评价与思考（深度见解）

### 1. 评测范式的转移：从“物理沙盒”到“逻辑仿真”
OccuBench 的成功证明了，对于复杂的专业领域，**语义逻辑的一致性**比**物理真实性**更重要。LES 巧妙地绕过了高昂的系统搭建成本，利用 LLM 内化的领域知识构建了“逻辑闭环”。这种“用 AI 评估 AI”的范式虽然存在潜在的“共识偏差”（即 Agent 可能只是学会了取悦模拟器），但在当前技术阶段，它是平衡评估深度与覆盖面的最优解。

### 2. 揭示了“静默失败”的防御真空
论文中关于隐式故障（E2）的测试极具实战价值。在真实生产环境中，API 极少直接崩溃，更多是返回格式不整齐或部分字段丢失的数据。实验显示当前最顶尖的 Agent 依然高度依赖“标准反馈”，缺乏对底层数据完整性的批判性观察。这预示着未来 Agent 的研发重点应从单纯的“指令遵循”转向“鲁棒性感知”和“异常诊断”。

### 3. 跨行业能力的“各具特色”
实验结果显示不同模型在行业表现上的“割裂感”引人深思。这说明现有的预训练语料分布在不同专业领域依然存在显著的厚薄差异。对于企业用户而言，选择 Agent 模型不应只看综合榜单（Leaderboard），而应参考类似 OccuBench 提供的“雷达图”，根据业务所属的行业细分（如金融 vs 农业）进行定制化选型。

---

## 适用场景
- **行业级 Agent 开发与选型**：在医疗、金融、工业制造等垂直领域进行 Agent 模型的能力评估。
- **Agent 鲁棒性压力测试**：在部署到真实 API 环境前，通过 OccuBench 的故障注入模式测试其容错与自愈能力。
- **推理成本优化研究**：研究在不同难度的专业场景下，推理成本（Thinking Tokens）与任务成功率之间的平衡点。

---

**总结**：OccuBench 不仅仅是一个基准测试，它为 AI Agent 走向真实职业世界提供了一套低成本、高效率的“全科模拟考场”。其引入的语言环境模拟（LES）和隐式鲁棒性维度，为 Agent 从“玩具”进化为“生产力工具”指明了关键的演进方向。
